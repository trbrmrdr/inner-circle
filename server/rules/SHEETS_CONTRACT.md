# Контракт Google Sheets

Этот файл - короткая точка сверки между:

- `tools/google-workspace/apps-script/Config.gs`
- `server/src/sheets/GoogleSheetsService.ts`
- `server/src/sheets/SheetsSchema.ts`
- `server/scripts/sync-google-sheets.ts`

Перед включением автопостинга можно проверить или выровнять таблицу командой из
папки `server`:

```bash
npm run sheets:check
npm run sheets:sync
```

Для копии проекта с другой таблицей:

```bash
npm run sheets:check -- --spreadsheet-id <sheet-id> --credentials <service-account-json>
npm run sheets:sync -- --spreadsheet-id <sheet-id> --posts-sheet POSTS --media-sheet MEDIA
```

Правило сетки: синхронизация создаёт недостающие листы/колонки, расширяет сетку
при необходимости и держит один пустой ряд/колонку сверх используемой области.
Данные, непустые строки и пользовательские колонки не удаляются. Для поведения
“только расширять” использовать `--no-trim-grid`.

Комментарии к заголовкам и описания настроек пишутся plain text: Google Sheets
notes не поддерживают жирное форматирование. Поэтому строгие значения и примеры
выделяются квадратными скобками, например `[ready]`, `[Telegram; VK; Instagram]`,
`[true]`, `[60000]`. Для читаемости использовать секции `ЗНАЧЕНИЯ`, `ПРИМЕР`,
`ФОРМАТ VALUE` и переносы строк. Для платформ регистр неважен.

## Листы

| Назначение | Лист |
| --- | --- |
| Очередь автопостинга | `POSTS` |
| Каталог медиа Google Drive | `MEDIA` |
| Заявки с сайта | `LEADS` |
| Логи сервера | `LOGS` |
| Настройки выполнения | `SETTINGS` |

## LEADS

| Поле | Обязательность | Описание |
| --- | --- | --- |
| `created_at` | да | Серверное время создания заявки. |
| `name` | нет | Имя из формы. |
| `phone` | контакт | Должен быть хотя бы один контакт: телефон, email или Telegram. |
| `email` | контакт | Должен быть хотя бы один контакт. |
| `telegram` | контакт | Опциональный `@username`. Проверяется только формат. |
| `date` | нет | Желаемая дата/день. |
| `guests` | нет | Количество гостей. |
| `scenario` | нет | Сценарий/комментарий из формы. |
| `consent` | нет | Согласие пользователя. |
| `meta_json` | нет | Компактный JSON с данными captcha/frontend. |

`GoogleSheetsService.AppendLead` пишет по названиям колонок. При первом лиде
после запуска процесса сервер проверяет `LEADS`, создаёт лист при отсутствии и
добавляет недостающие серверные колонки без удаления и перестановки существующих.

Телефон записывается как `RAW`, чтобы Google Sheets не превращал его в число или
экспоненциальную запись. `sheets:sync` форматирует колонку `LEADS.phone` как
обычный текст и приводит старые числовые телефоны к виду `+7 999 000-00-00`, если это
можно сделать безопасно.

## POSTS

| Поле | Обязательность | Описание |
| --- | --- | --- |
| `post_id` | да | Внутренний уникальный ключ поста. Генерируется Apps Script для незапечатанных строк. |
| `date` | да | Дата плановой публикации. |
| `time` | да | Время плановой публикации. |
| `status` | да | Ручной gate автопостинга. Сервер публикует только `ready`, затем пишет `processing`, `posted`, `partial`, `error`, `skipped`. |
| `text` | контент | Исходный текст до обработки DeepSeek. Может быть пустым, если есть `media_ids`. |
| `platforms` | да | Список через запятую/точку с запятой/перенос: `telegram`, `vk`, `instagram`, `facebook`. Алиасы: `tg`, `ig`, `inst`, `fb`, `вк`. |
| `media_ids` | контент | Список ID из `MEDIA.media_id` через запятую/точку с запятой/перенос. Может быть пустым, если есть `text`. |
| `telegram_status` | нет | Состояние Telegram-публикации: `processing`, `posted`, `error`, `skipped`, `disabled`. |
| `telegram_lock_until` | нет | Временный lock Telegram-публикации, чтобы другой запуск не взял тот же пост. |
| `telegram_published_at` | нет | Время успешной публикации в Telegram. |
| `telegram_message_id` | нет | Заполняется после успешной публикации в Telegram. |
| `telegram_url` | нет | Ссылка на Telegram-пост, если её можно собрать из chat id и message id. |
| `telegram_error` | нет | Последняя ошибка Telegram-публикации. |
| `telegram_response` | нет | Компактный JSON ответа Telegram. |
| `vk_status` | нет | Состояние VK-публикации: `processing`, `posted`, `error`, `skipped`, `disabled`. |
| `vk_lock_until` | нет | Временный lock VK-публикации, чтобы другой запуск не взял тот же пост. |
| `vk_published_at` | нет | Время успешной публикации во ВКонтакте. |
| `vk_post_id` | нет | Заполняется после успешной публикации во ВКонтакте. |
| `vk_url` | нет | Ссылка на VK-пост вида `https://vk.com/wall-<group_id>_<post_id>`. |
| `vk_error` | нет | Последняя ошибка VK-публикации. |
| `vk_response` | нет | Компактный JSON ответа VK. |

Публикация выбирается только для строк со `status=ready`. Дополнительно
обязательны `post_id`, `date`, `time`, `platforms` и хотя бы одно поле контента:
`text` или `media_ids`. Если `status` пустой, `template` или `draft`, строка не
публикуется, даже если дата и платформы заполнены. Если `platforms` пустой,
сервер не подставляет Telegram по умолчанию и строка игнорируется.

Поддерживаемые статусы:

- пусто - строка не публикуется;
- `template` - шаблон, не публикуется;
- `draft` - черновик, не публикуется;
- `ready` - можно публиковать;
- `processing` - сервер взял строку в работу;
- `posted` - публикация прошла успешно;
- `partial` - часть платформ успешна, часть нет;
- `error` - ошибка, для повтора вручную вернуть `ready`;
- `skipped` - пропущено, для повтора вручную вернуть `ready`.

Apps Script управляет `post_id` так:

- если `date`, `time`, `platforms` и хотя бы один контентный блок заполнены, для незапечатанной строки генерируется ID вида `YYMMDD-HHMM-HASH-T`, где последний блок показывает платформы;
- если обязательные поля очищены, `post_id` тоже очищается;
- если изменились `date`, `time`, `platforms`, `text` или `media_ids`, ID незапечатанной строки меняется;
- если статус `processing`, `posted`, `done`, `partial` или уже есть platform message/post id, строка запечатана и ID не меняется;
- дубли и отсутствующие ID в готовых/запечатанных строках подсвечиваются красным.

`*date_marker` показывает календарь: прошлое красное, сегодня зеленое, будущее
синее. Если строка со `status=ready` невалидна для публикации, маркер становится
фиолетовым.

Настройки выполнения:

| Настройка | Описание |
| --- | --- |
| `autopost.enabled` | Включает автопостинг в Google таблице. Env `AUTOPOST_ENABLED` тоже должен быть `true`. |
| `autopost.interval_ms` | Частота проверки таблицы. |
| `autopost.publish_window_minutes` | Сколько минут после планового времени пост еще можно выпустить, если сервер был выключен. |
| `autopost.future_grace_seconds` | Небольшой допуск вперед относительно текущей проверки. |
| `deepseek.enabled` | Включает подготовку текста через DeepSeek. Env `DEEPSEEK_ENABLED` тоже должен быть `true`. |
| `deepseek.telegram.prompt` | Английский Telegram prompt для DeepSeek. Если `value` пустой, сервер использует default prompt из кода. |
| `deepseek.vk.prompt` | Английский VK prompt для DeepSeek. Если `value` пустой, сервер использует default prompt из кода. |

`SETTINGS.value` - рабочее значение, которое читает сервер.
`SETTINGS.description` - русское описание формата и поведения.
`SETTINGS.value_ru` - русская версия значения или prompt для понимания; сервер это
поле не читает.

Если `SETTINGS.value` пустой или ключа нет, сервер использует default текущего
env/кода. Это обязательное поведение: пустая настройка не должна ломать
автопостинг.

## MEDIA

| Поле | Обязательность | Описание |
| --- | --- | --- |
| `media_id` | да | Ключ, который вставляется в `POSTS.media_ids`. |
| `type` | желательно | `image` или `video`; используется как подсказка для человека и сервера. |
| `mime_type` | желательно | MIME исходного файла. Нужен для корректной подготовки Telegram media. |
| `file_id` | желательно | Google Drive file ID для скачивания приватного файла сервисным аккаунтом. |
| `preview_url` | нет | URL превью для таблицы. Для видео это созданная preview-картинка. Не используется как источник публикации. |
| `drive_url` | нет | Ссылка на файл в Google Drive. |
| `public_url` | для Meta | Публичный HTTPS URL для Instagram/Facebook. |
| `media_url` | нет | Прямой URL медиа, если он есть. |

Рабочий формат `media_id`:

```text
IMG_001, IMG_002, VID_0001, VID_0002
```

Префикс нужен для читаемости в таблице. Сервер не должен определять реальный тип
медиа только по `media_id`; для публикации используются `type`, `mime_type` и
скачанный файл.

Для Instagram и Facebook медиа должно быть доступно по публичному HTTPS URL.
Приватные Google Drive файлы подходят для Telegram-пайплайна, но не подходят для
Meta Graph API.

До полной реализации Instagram/Facebook серверная синхронизация не добавляет
отдельные runtime-колонки и DeepSeek prompt-строки для этих сетей. Новая сеть
сначала получает publisher, media pipeline, platform runtime-колонки, тестовую
команду и rules, и только после этого добавляется в управляемые настройки таблицы.

Для VK и Telegram приватные Google Drive файлы скачиваются сервисным аккаунтом на
текущий сервер, затем конвертируются в платформенную временную папку. `preview_url`
для публикации не используется.

## Правки данных таблицы

Одноразовые массовые изменения Google Sheets не добавляются в Apps Script. Для
миграций, переименования ID, исправления строк и похожих задач пишется локальный
временный скрипт под конкретную задачу, запускается через Sheets API с сервисным
аккаунтом, проверяется в режимах dry-run/apply, после чего временный скрипт
удаляется.

В Apps Script оставляем только постоянные функции таблицы и Google Drive:
синхронизация `MEDIA`, превью, сортировка, usage, удаление выбранных медиа.

## Telegram test autopost

Локальные команды из папки `server`:

```bash
npm run autopost:telegram:test
npm run autopost:telegram:test -- --post-id 0000a --prepare-only --yes
npm run autopost:telegram:test -- --post-id 0000a --yes
```

Правила теста:

- читает `POSTS` и `MEDIA` один раз за запуск;
- принимает любой статус поста: `draft`, `ready`, `posted` и т.д.;
- скачивает приватные Google Drive файлы сервисным аккаунтом в `server/tmp/autopost/<post_id>/<run_id>/source`;
- готовит Telegram-файлы в `server/tmp/autopost/<post_id>/<run_id>/telegram`;
- не использует `MEDIA.preview_url` для публикации;
- пропускает missing/битые/неподдержанные media item и продолжает постить оставшийся текст/медиа;
- падает только если после подготовки не осталось ни текста, ни медиа;
- после успешной реальной Telegram-публикации удаляет папку конкретного запуска из `tmp`;
- при ошибке оставляет папку запуска и `manifest.json` для диагностики;
- фото всегда отправляются как JPEG media, не как документы;
- видео приводятся к MP4/MPEG4 через `scripts/media/telegram-video-normalize.sh`, если это нужно;
- для 2-10 медиа используется один `sendMediaGroup`, включая смешанные альбомы фото + видео;
- текст поста прикрепляется как caption первого медиа, потому что общего caption для всего альбома в Bot API нет;
- Telegram сам выбирает сетку альбома, сервер не управляет размером ячеек в клиенте;
- DeepSeek готовит Telegram caption под лимит 1024 символа для медиа-постов и 4096 символов для text-only;
- `sendDocument` для автопостинга запрещён;
- пишет `manifest.json` в папку запуска;
- отправляет только в `TELEGRAM_TECH_CHAT_ID`;
- Google Sheets в тестовом режиме не обновляет.

`--prepare-only` только скачивает и готовит медиа, но не отправляет сообщение в
Telegram.

## VK test autopost

Локальные команды из папки `server`:

```bash
npm run vk:group-id -- --group <short_name_or_url>
npm run autopost:vk:test
npm run autopost:vk:test -- --post-id 0000a --prepare-only --yes
npm run autopost:vk:test -- --post-id 0000a --yes
```

Правила теста:

- читает `POSTS` и `MEDIA` один раз за запуск;
- принимает любой статус поста: `draft`, `ready`, `posted` и т.д.;
- скачивает приватные Google Drive файлы сервисным аккаунтом в `server/tmp/autopost/<post_id>/<run_id>/source`;
- готовит VK-файлы в `server/tmp/autopost/<post_id>/<run_id>/vk`;
- не использует `MEDIA.preview_url` для публикации;
- пропускает missing/битые/неподдержанные media item и продолжает постить оставшийся текст/медиа;
- падает только если после подготовки не осталось ни текста, ни медиа;
- после успешной реальной VK-публикации удаляет папку конкретного запуска из `tmp`;
- при ошибке оставляет папку запуска и `manifest.json` для диагностики;
- фото всегда отправляются как VK photo attachment, не как документы;
- видео приводятся к MP4/H.264/AAC через `scripts/media/vk-video-normalize.sh`;
- текст поста идет в `wall.post.message`, медиа идут в `wall.post.attachments`;
- Google Sheets в тестовом режиме не обновляет.

`VK_GROUP_ID` хранится положительным числом. Отдельный owner env не используется:
для `wall.post.owner_id` сервер сам отправляет отрицательный ID группы.
